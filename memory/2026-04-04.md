# Zeus 审计报告 - 2026-04-04

## 假设与现实不一致

### 1. Bin 结构（部分核实）
- **假设**: `fahrenheit_width=2`，格式 "50-51°F"（覆盖 2 个整数值）
- **实际**: positions-paper.json 中 bin_label 显示 "between 52-53°F"、"between 54-55°F" 等，确实每个 bin 覆盖 2 个整数
- **结论**: ✅ 与假设一致（F 城市）
- **注意**: Gamma API 无法返回天气市场（active=true 的 100 个市场中 0 个为天气市场），bin 结构无法从 API 侧独立核实

### 2. MC counts
- **假设**: `mc_count_entry=5000`，`mc_count_monitor=5000`
- **实际**: `config/settings.json` 确认 `ensemble.n_mc=5000`，`day0.n_mc=5000`，`instrument_noise_f=0.5`，`instrument_noise_c=0.28`
- **结论**: ✅ 数值与假设一致

### 3. 信号噪声
- **假设**: `instrument_noise_f=0.5`，`instrument_noise_c=0.28`
- **实际**: settings.json 确认相同数值
- **结论**: ✅ 一致

### 4. Settlement 精度与 rounding（重大不一致）
- **假设**: `precision_f=1.0`，`rounding_rule=round_half_to_even`，settlement 来源 WU
- **实际**: `state/wu_settlement_audit.json` 显示：
  - **总体 mismatch rate = 40.7%**（1385 条记录中 564 条不一致）
  - London：**76.2% mismatch**（324/429）——极严重
  - Los Angeles：50% mismatch（4/8）
  - San Francisco：50% mismatch（3/6）
  - Paris：31.8% mismatch（14/44）
  - NYC：35.7% mismatch（150/420）
  - match rate 仅 30.8%（427/1385）
- **结论**: ❌ 严重不一致——假设认为 WU 结算规律稳定，但实际 mismatch rate 40.7%，London 高达 76%

### 5. 数据库状态（未预期发现）
- `zeus.db`、`trading.db`、`risk_state.db`、`zeus_state.db` 全部无任何表（空数据库）
- 真实持仓数据存储在 JSON 文件（`positions-paper.json`）中
- `positions.json` 已被废弃归档，指向 positions-paper.json / positions-live.json
- **结论**: 数据库层尚未初始化或已清空，这影响 `_check_persistence_anomaly()` 等依赖 DB 查询的逻辑

## 代码路径一致性检查

### MC Counts 不一致 ⚠️

**evaluator.py（entry 路径）**:
```python
# line 403 — ENS 路径，无 n_mc 参数
p_raw = ens.p_raw_vector(bins)

# line 397 — Day0 路径，无 n_mc 参数
p_raw = day0.p_vector(bins)
```

**monitor_refresh.py（monitor 路径）**:
```python
# line 83 — ENS 路径，显式传 n_mc
p_raw_single = float(ens.p_raw_vector(single_bin, n_mc=ensemble_n_mc())[0])

# line 218 — Day0 路径，显式传 n_mc
p_raw_yes = float(day0.p_vector(single_bin, n_mc=day0_n_mc())[0])
```

**差异**: evaluator.py 的 entry 路径不传 `n_mc`，依赖方法内部默认值；monitor_refresh.py 显式传 `ensemble_n_mc()` / `day0_n_mc()`。
如果 `p_raw_vector` 的默认 n_mc 与 `ensemble_n_mc()` 返回值不同，则 entry 和 monitor 使用不同的 MC 次数，违反 Blueprint v2 §7 "same method as entry" 原则。

semantics = SettlementSemantics.for_city(city)

### Bin Handling 不一致 ⚠️

**evaluator.py（entry）**:
```python
# 从 market scanner outcomes 直接读取 range_low / range_high
low, high = o["range_low"], o["range_high"]
bins.append(Bin(low=low, high=high, label=o["title"], unit=city.settlement_unit))
```

**monitor_refresh.py（monitor）**:
```python
# 从 position.bin_label 字符串重新解析
low, high = _parse_temp_range(position.bin_label)
```

**差异**: Entry 路径从 Gamma API 直接读取数值边界；monitor 路径从 bin_label 文字重新解析。
如果 `_parse_temp_range` 解析逻辑与 Gamma API 提供的 `range_low/range_high` 数值不严格对应（例如格式变化、边界含义差异），可能导致 monitor 计算 bin 与 entry 不一致。

## 警示

### P0 — Settlement 数据严重不可靠
- WU settlement audit mismatch rate 40.7%，London **76%**
- 当前所有 14 个 paper positions 全部是 `buy_no` 方向（注意：没有 buy_yes），且全部是 ENS entry 方法
- 如果 settlement 数据无法可靠预测，整个交易逻辑的 P/L 计算基础存疑
- **需要立即**：检查 wu_settlement_audit.json 中具体的 mismatch 原因（时区？数据源变更？rounding 规则不同于 round_half_to_even？）

### P1 — 数据库全部为空
- `_check_persistence_anomaly()` 查询 `settlements` 和 `temp_persistence` 表，均不存在
- 该函数有 try/except 保护，失败时返回 `1.0`（无折扣）
- 实际上 persistence anomaly check 从未生效，entry 和 monitor 的 anomaly_discount 始终为 1.0
- 所有基于历史数据的 DB 查询（model_bias, strategy_tracker 等）均无效

### P2 — MC count 参数不一致（entry vs monitor）
- evaluator.py 中 `p_raw_vector(bins)` 未传 n_mc，可能使用不同默认值
- 需确认 `EnsembleSignal.p_raw_vector` 和 `Day0Signal.p_vector` 的默认 n_mc 是否等于 `ensemble_n_mc()` / `day0_n_mc()`

### P3 — Bin 解析路径双轨
- Entry 用 `range_low/range_high`，monitor 用 `_parse_temp_range(bin_label)`
- 测试需验证两种路径对所有 bin 格式（shoulder bins、标准 2°F bins）的一致性
