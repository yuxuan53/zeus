# Zeus 每日审计报告 — 2026-04-09

## 一、结论摘要

`state/assumptions.json`（上次更新 2026-03-31）存在 **4 项与现实不一致的假设**，其中 2 项为高风险。`evaluator.py` 与 `monitor_refresh.py` 的核心信号路径（MC 计数、bin 处理、settlement 语义）基本一致，但存在 1 处语义漂移和若干残留代码问题。

---

## 二、假设与现实不一致项

### 2.1 [高风险] Settlement 精度假设与实际数据不符

**假设**: `precision_f: 1.0`, `precision_c: 1.0`（即整数结算）

**现实**:
- **°F 城市**: 近期（2026-04 起）大量使用 `openmeteo_archive_harvester` 作为结算源，产出小数值（如 NYC 69.1, Chicago 55.4, Miami 80.5, Atlanta 81.2）。这些值 **不是整数**。
- **°C 城市**: Seoul 和 Tokyo 使用 `openmeteo_archive_daily_max`，产出小数值（如 Tokyo 21.6, 16.4; Seoul 19.5, 15.7）。London/Paris/Shanghai 的 WU 数据仍为整数。
- 即便 WU 来源的历史 °F 数据也非整数（NYC wu_daily_observed: 50.7, 102.4）。

**影响**: `SettlementSemantics` 代码中 `precision=1.0` 会将 MC 模拟结果四舍五入到整数后分配到 bin。如果真实结算是小数（如 69.1°F），则 bin 归属判定与实际不同。整数假设下 69.1 → 69，可能落入 68-69°F bin，但若结算值就是 69.1，则应落入 68-69 bin（因为 ≤69）或 70-71 bin（取决于 bin boundary 定义）。**这是概率计算与结算判定之间的系统性偏差源。**

**根因**: Settlement 源从 WU 切换到 openmeteo 后，precision 假设未更新。

### 2.2 [高风险] Settlement 数据源假设过时

**假设**: `source: "Weather Underground (WU)"`

**现实**: 实际结算源已多样化：
| 源类型 | 覆盖城市 |
|--------|----------|
| `openmeteo_archive_harvester` | NYC, Chicago, Atlanta, Miami, Dallas, Austin, Houston, Seattle, LA, SF, Denver |
| `openmeteo_archive_daily_max` | Seoul, Tokyo |
| `wu_icao_*` | London, Paris, Shanghai（仍为 WU）|
| `wu_daily_observed` | 历史回填数据 |
| `wu_icao_history` | Buenos Aires |

**影响**: 代码中 Day0Signal 的注释仍写着 "Observation is known (integer °F/°C from WU)"，但实际 openmeteo 数据非整数。settlement chain 假设（`Atmosphere → NWP → ASOS → METAR → WU display → Settlement`）对 openmeteo 源不适用。**如果 Polymarket 仍以 WU 整数结算，但我们存储的是 openmeteo 小数，则回测和 persistence anomaly 检查的 settlement_value 基准就是错误的。**

### 2.3 [中风险] 城市列表不完整

**假设**: 11 个 °F 城市 + 5 个 °C 城市 = 16 个城市

**现实**: settlements 表中包含 21 个城市，多出：
- Buenos Aires（°C）
- Hong Kong（°C）  
- Munich（°C）
- Shenzhen（°C）
- Wellington（°C）

这些城市有结算数据但不在 assumptions.json 的城市清单中。

**影响**: 若这些城市参与交易但 assumptions 未覆盖其 bin 结构/settlement 语义，信号管线可能使用默认参数而非城市特定参数。

### 2.4 [低风险] Ensemble 成员数假设基本正确

**假设**: `ens_member_count: 51`

**现实**: `ensemble_snapshots` 表中 `json_array_length(members_json) = 51`，最近 5 条记录均为 51。

**结论**: 一致，无问题。

### 2.5 [低风险] MC 计数一致

**假设**: `mc_count_entry: 5000`, `mc_count_monitor: 5000`

**现实**: `ensemble_n_mc()` = 5000, `day0_n_mc()` = 5000。evaluator 和 monitor_refresh 均使用这些配置函数。

**结论**: 一致，无问题。

## 三、代码路径一致性检查（evaluator.py vs monitor_refresh.py）

### 3.1 MC 计数

| 路径 | ENS MC | Day0 MC |
|------|--------|---------|
| evaluator (entry) | `ens.p_raw_vector(bins)` → 默认 `ensemble_n_mc()` = 5000 | `day0.p_vector(bins)` → 默认 `day0_n_mc()` = 5000 |
| monitor_refresh | `ens.p_raw_vector(single_bin, n_mc=ensemble_n_mc())` | `day0.p_vector(single_bin, n_mc=day0_n_mc())` |

**结论**: 一致。monitor 显式传入 n_mc 参数，evaluator 使用默认值，但默认值与显式值相同。

### 3.2 Settlement 语义

| 路径 | 语义获取 |
|------|----------|
| evaluator | `SettlementSemantics.for_city(city)` → 传入 EnsembleSignal |
| monitor ENS | `SettlementSemantics.for_city(city)` → 传入 EnsembleSignal |
| monitor Day0 | 未显式获取 SettlementSemantics，依赖 Day0Signal 内部处理 |

**结论**: 基本一致。Day0 路径在两侧都依赖 Day0Signal 内部的 `_settle()` 方法。

### 3.3 Bin 处理

| 路径 | Bin 构造 |
|------|----------|
| evaluator | 从 outcomes 解析完整 bins 列表，`Bin(low=low, high=high, label=o["title"], unit=city.settlement_unit)` |
| monitor | 从 position 的 `bin_label` 解析单个 bin，`[Bin(low=low, high=high, label=position.bin_label, unit=position.unit)]` |

**潜在漂移**: evaluator 使用 `city.settlement_unit`，monitor 使用 `position.unit`。如果 position 存储时 unit 与 city config 不一致，可能产生偏差。实际上 position.unit 应来自入场时的 city.settlement_unit，所以正常情况下一致。

### 3.4 [语义漂移] Alpha 计算中的 hours_since_open

| 路径 | hours_since_open |
|------|------------------|
| evaluator | `candidate.hours_since_open`（来自 market scanner，表示市场开放后的小时数）|
| monitor | 从 `position.entered_at` 计算实际持仓时间，默认 fallback 48h |

**漂移说明**: evaluator 的 `hours_since_open` 是市场维度的（"市场开放多久了"），monitor 的是仓位维度的（"这个仓位持有多久了"）。这两个语义不同：
- 市场可能开放 72h，但仓位可能只持有 2h
- `compute_alpha` 对 `hours_since_open` 的响应决定了模型信心权重

这意味着 **同一个仓位在 entry 和 monitor 时，alpha 的 hours 分量语义不同**。entry 时基于市场年龄，monitor 时基于仓位年龄。这可能导致 monitor 阶段的 alpha 系统性偏高或偏低。

### 3.5 残留代码问题

两个文件中均存在重复的 Semantic Provenance Guard 死代码：
```python
# Semantic Provenance Guard
# Semantic Provenance Guard
if False: _ = None.selected_method; _ = None.entry_method
if False: _ = None.selected_method; _ = None.entry_method
```
每个关键函数中重复两次。这是纯死代码，不影响运行，但增加阅读噪音。

## 四、需要警示的风险点

1. **🔴 Settlement 精度/来源漂移 — 最高优先级**  
   openmeteo 回填的小数 settlement 值与 WU 整数假设冲突。如果 Polymarket 仍以 WU 整数结算，则 DB 中的 settlement_value 不等于真实结算值。这影响：
   - `_check_persistence_anomaly()` 中的 delta 计算
   - 所有回测和 calibration_pairs 的真值基准
   - forecast_skill 评估

2. **🟡 hours_since_open 语义漂移**  
   Entry 用市场年龄、monitor 用仓位年龄计算 alpha，可能导致 monitor 时 alpha 不连续（entry 时 hours=72，首次 monitor 时 hours=0.5）。

3. **🟡 未覆盖城市参与交易**  
   Buenos Aires、Hong Kong 等 5 个城市有 settlement 数据但不在 assumptions.json 城市列表中。若它们进入交易管线，settlement 语义可能使用错误默认值。

4. **🟢 assumptions.json 更新日期陈旧**  
   `updated_at: "2026-03-31"` — 已 9 天未更新。建议在修复上述问题后同步更新。
