# Zeus 日常审计 — 2026-04-08

## 一、assumptions.json 核对结果

### 1. Settlement 精度假设不一致

**假设**: `precision_f: 1.0`, `precision_c: 1.0`（整数精度），`settlement_chain` 描述 WU 显示整数。

**现实**: zeus.db `settlements` 表中 `settlement_value` 存储的值均为 1 位小数（如 57.2、53.1、87.1、46.6）。

- **证据**: `SELECT settlement_value FROM settlements WHERE settlement_value IS NOT NULL ORDER BY target_date DESC LIMIT 15` 全部为 1 位小数。
- **影响**: assumptions.json 声称 WU settlement 是整数，但实际存储值带小数。代码中 `_simulate_settlement` 确实做了 round-to-integer，所以信号计算本身是正确的。但 **settlement_value 列存储的不是最终 settlement 值，而是 WU 观测原始值**（可能带 0.1 精度），这导致 persistence_anomaly_check 中 `predicted_high - row["settlement_value"]` 计算的 delta 基于非整数值，与假设矛盾。

### 2. Settlement 数据收集断裂

**现实**: 自 2026-03-29 起，大量 settlement 记录 `settlement_value = NULL`（共 9 条），但 `winning_bin` 仍有值。

- **证据**: NYC/Atlanta/Miami/Dallas/Chicago 的 3/29 和 NYC/Paris 的 3/30 记录均无 `settlement_value`。
- **影响**: `_check_persistence_anomaly()` 在 monitor_refresh.py 中依赖近 3 天的 settlement_value 做历史异常检测。NULL 值导致这个安全机制失效——无法检测到异常温度变化。

### 3. Settlement value 与 winning_bin 不一致

**现实**: 多个 F 城市的 settlement_value 与 winning_bin 不匹配：
- NYC 3/28: val=50.7 → round=51，但 bin=42-43（差 8-9°F）
- Atlanta 3/28: val=69.1 → round=69，但 bin=70-71
- Miami 3/28: val=82.6 → round=83，但 bin=84-85

**影响**: settlement_value 和 winning_bin 来自不同数据源或不同时间点。settlement_value 可能是 ETL 采集的当时观测值（非最终日最高温），而 winning_bin 来自 Polymarket 的官方结算。这意味着 settlement_value 列作为 truth source 不可靠。

### 4. MC 计数 — 一致

**假设**: `mc_count_entry: 5000`, `mc_count_monitor: 5000`。
**现实**: `config/settings.json` 中 `ensemble.n_mc = 5000`, `day0.n_mc = 5000`。`p_raw_vector()` 和 `p_vector()` 默认均调用对应配置值。**一致，无问题**。

### 5. ENS 成员数 — 一致

**假设**: `ens_member_count: 51`。
**现实**: `validate_ensemble()` 默认检查 51 成员。**一致**。

## 二、monitor_refresh.py vs evaluator.py 语义一致性

### 发现 1：校准路径分歧（中等风险）

| 维度 | evaluator.py（entry） | monitor_refresh.py（monitor） |
|------|----------------------|------------------------------|
| 校准方式 | `calibrate_and_normalize(p_raw, cal, lead_days, bin_widths)` — 全 bin 向量校准+归一化 | `cal.predict_for_bin(p_raw_single, lead_days, bin_width)` — 单 bin 校准，无跨 bin 归一化 |
| 影响 | 概率向量 sum=1.0 | 单 bin 概率可能偏高或偏低 |

**评估**: entry 路径对整个概率向量做 Platt 校准后归一化确保 sum=1，monitor 路径只对单个 bin 做校准。当 Platt 模型对各 bin 的拉伸/压缩不均匀时，单 bin 校准后的概率可能与全向量校准后的对应 bin 概率有差异。这种差异会导致 monitor 的 p_posterior 相对 entry 时的 p_posterior 产生系统性漂移。

### 发现 2：Semantic Provenance Guard 重复死代码

两个文件均包含重复的 Semantic Provenance Guard 代码块：
```python
# Semantic Provenance Guard
# Semantic Provenance Guard
if False: _ = None.selected_method; _ = None.entry_method
if False: _ = None.selected_method; _ = None.entry_method
```

每处出现 2 次（4 行），且是 `if False:` 死代码。monitor_refresh.py 中出现 3 处（共 12 行），evaluator.py 中出现 2 处（共 8 行）。虽然不影响运行时行为，但增加代码噪音。

### 发现 3：Day0 路径 — 一致

- evaluator 的 Day0 路径和 monitor 的 `_refresh_day0_observation` 使用相同的 `Day0Signal` 构造和 `p_vector()` 调用。
- 两者均使用 `remaining_member_maxes_for_day0()` 和 `build_day0_temporal_context()`。
- MC count 默认值一致（`day0_n_mc() = 5000`）。
- **Day0 路径语义一致**。

### 发现 4：Alpha 计算 — 一致

- 两个路径均使用 `compute_alpha(calibration_level, ensemble_spread, model_agreement, lead_days, hours_since_open)`。
- Monitor 路径从 position 记录计算实际 hours_since_open（非硬编码）。
- **一致**。

## 三、摘要与警示

### 警示
1. **settlement_value 列不可信** — 值与 winning_bin 不匹配，精度与假设矛盾，且近期大量 NULL。任何依赖此列的逻辑（persistence_anomaly_check）实质上已失效。
2. **校准路径分歧** — entry 和 monitor 使用不同的校准策略（全向量 vs 单 bin），可能导致 monitor 阶段概率漂移，影响退出决策准确性。
3. **数据采集断裂** — 自 3/29 起 settlement_value 停止填充，ETL 管道可能有故障。

### 建议修复方向
1. **修复 settlement ETL**: 排查 `wu_daily_collector` 为何自 3/29 起不再填充 settlement_value。优先恢复数据采集。
2. **统一校准路径**: monitor_refresh 应使用与 evaluator 相同的 `calibrate_and_normalize` 全向量方法（构造完整 bin 列表后取目标 bin 的校准概率），或在 assumptions.json 中明确记录两条路径的设计意图差异。
3. **清理死代码**: 删除重复的 Semantic Provenance Guard 死代码块。
4. **明确 settlement_value 语义**: 在 DB schema 或文档中明确此列存储的是 "WU 观测原始值" 还是 "最终 settlement 温度"，并确保 assumptions.json 的 precision 声明与实际一致。
