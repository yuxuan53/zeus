# 01 — Plan Delta vs Current Repo

The uploaded plan is directionally correct but stops short of the deeper failure: Zeus has not only a noise problem; it has a **cognition extraction problem**. The plan says topology should remain the governance/routing kernel and become scope-aware, advisory-first, typed, fixture-backed, repairable, and graph-aware. That remains right. The repo evidence adds a stronger requirement: topology must rehydrate system knowledge into durable module/system books.

## Ideas from the plan that remain correct

- Topology should not be deleted.
- Global health should be separated from scoped closeout.
- Navigation should generate useful route/context even when unrelated drift exists.
- Issues need richer machine metadata.
- Live repo-health tests should not make fixture tests flaky.
- Graph should be useful but non-authoritative.

## Ideas that need stronger framing

### “Lower noise” is not enough

A quieter topology doctor can still be cognitively hollow. The target must be **scope-aware and knowledge-rich**, not merely less noisy.

### “More manifests” is not enough

The repo already has many manifests. Adding more manifest entries without ownership normalization will make drift worse.

### “Module books” are not optional documentation

For online-only agents, module/system books are the bridge between compressed machine registries and Zeus-level reasoning. They should be treated as a first-class cognition layer, not a nice-to-have.

## Missing deeper implication

The plan talks about typed issues, repair drafts, and graph hardening. The deeper implication is:

> Every recurring topology_doctor rule, test fixture law, packet-plan convention, graph-derived route, and manifest ownership decision needs a durable home: either authority, reference cognition, machine manifest data, or explicitly non-authoritative evidence.

Until that extraction happens, Zeus will keep rediscovering hidden obligations manually.
