# Not Now — Explicit Deferrals

These items are intentionally deferred. They are not rejected forever; they are rejected **at this stage** because Zeus still needs durable execution truth first.

## 1. More complex posterior / calibration / hierarchical Bayesian work

Not now.

### Why deferred
Better posterior quality does not fix orphan command windows, restart ambiguity, degraded authority inflation, or unresolved execution truth.

### Prerequisite
P1 and P2 complete, with durable command truth and unknown-state handling verified.

---

## 2. Queue-position / impact / slicing simulator

Not now.

### Why deferred
The repo does not yet have a durable command-event spine and authoritative user-order/trade update handling. Simulating queue/impact before the event ground truth exists would optimize against imaginary execution semantics.

### Prerequisite
P1 command journal, P2 semantics, and stable V2 event ingestion.

---

## 3. Fake-capability execution labeling

Not now.

### Why deferred
Labels such as iceberg, dynamic peg, or liquidity guard should not multiply unless there is a real state machine and evidence surface behind them.

### Prerequisite
Gateway-only command architecture plus durable execution-state telemetry.

---

## 4. Automated quarantine/self-heal logic

Not now.

### Why deferred
Auto-heal before `UNKNOWN` / `REVIEW_REQUIRED` authority is mature risks automating false certainty.

### Prerequisite
P2 complete and operator-reviewed recovery paths proven.

---

## 5. Broad market expansion or aggressive eligibility loosening

Not now.

### Why deferred
Venue truth and settlement containment are not yet strong enough to justify expanding live surface area.

### Prerequisite
P3 eligibility policy complete and execution truth stable.

---

## 6. Broad stale test/doc cleanup unrelated to authority

Not now.

### Why deferred
P0 should clean only stale artifacts that directly mislead truth/authority implementation. A repo-wide documentation cleanup would expand scope without reducing the highest-risk live failure class.

### Prerequisite
P0/P1/P2 closeout, then a separate documentation hygiene packet.

---

## 7. Multi-venue abstraction

Not now.

### Why deferred
Zeus does not yet have a fully durable command truth model even for one venue. Abstracting multiple venues before that would generalize an incomplete authority boundary.

### Prerequisite
Single-venue execution truth stable and verified.

---

## 8. Rich operator UX redesign

Not now.

### Why deferred
The near-term need is truthful status, not visual polish. The operator must be able to see unresolved truth and degraded authority; that does not require a broad dashboard redesign yet.

### Prerequisite
P0/P2 state surfaces frozen.

---

## 9. Pulling P3 into P0/P1

Not now.

### Why deferred
Market eligibility and persistent alpha budget matter, but they should not be used as substitutes for incomplete command truth.

### Prerequisite
P1/P2 complete.

---

## 10. Re-enabling unrestricted live entry early

Not now.

### Why deferred
The system still has unresolved execution truth risk and near-term V2 venue migration risk.

### Prerequisite
P0, P1, and P2 complete; unresolved command count zero; V2 preflight verified; rollback posture tested.

## Summary rule

Anything that makes Zeus **sound** smarter without first making Zeus **more truthful about execution state** is deferred.
