## Initial critic: Laplace the 2nd

Initial verdict: BLOCK

Blocking issue:
- `ExitMutex.acquire()` was not atomic when reacquiring a previously released mutex row. Two concurrent replacements could both observe the released row and both return `True`, with the later writer stealing the holder.

Evidence from initial critic:
- Non-atomic released-row path selected `command_id, released_at`, then updated by `mutex_key` without a stale-row compare.
- Existing tests only covered sequential active-holder blocking and sequential reuse after release.

Remediation:
- `src/execution/exit_safety.py` now reacquires released rows with a compare-and-swap on the observed `(mutex_key, command_id, released_at)`.
- Stale compare failures return `False` unless the same command is already the active holder.
- Added regression: `tests/test_exit_safety.py::test_mutex_reacquire_released_row_fails_closed_on_stale_compare`.
