# G1 boot — live-readiness gates

Status: IN_PROGRESS
Date: 2026-04-27
Owner: Codex

## Objective

Implement the R3 G1 readiness surface that proves Zeus is *engineering-ready*
for operator-controlled live deployment only when:

1. all 17 explicit readiness gates pass, and
2. staged-live-smoke evidence proves the same 17/17 suite passed in a staged
   environment.

Even a green G1 report is not deployment authority. The blocking operator gate
remains `live-money-deploy-go`.

## Scoped surfaces

- `scripts/live_readiness_check.py`
- `tests/test_live_readiness_gates.py`
- `architecture/script_manifest.yaml`
- `architecture/test_topology.yaml`
- `architecture/topology.yaml`
- `tests/test_digest_profile_matching.py`
- R3 evidence/status/receipt documents under this packet.

## Non-goals and hard boundaries

- No live submit/cancel/redeem side effects.
- No production DB or runtime state mutation.
- No credential activation or live strategy promotion.
- No CLOB cutover or `LIVE_ENABLED` transition.
- No automatic execution of live smoke scripts; G1 reads staged-smoke evidence
  only and fails closed when evidence is absent.

## Proof questions

1. Does every readiness gate map to a concrete upstream antibody or structural
   scan? — Yes, the gate registry has exactly 17 IDs (`G1-01`..`G1-17`) and each
   includes phase/antibody metadata.
2. Does missing Q1 or staged-smoke evidence fail closed? — Yes, missing evidence
   returns `FAIL` and exit code 1.
3. Can the script authorize live deployment? — No. `live_deploy_authorized` is
   always false and the output states operator `live-money-deploy-go` is still
   required.
4. Does G1 run live side-effect smoke automatically? — No. It reads evidence
   files; it never invokes `scripts/live_smoke_test.py`.
